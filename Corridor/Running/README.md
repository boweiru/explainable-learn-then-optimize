
# Urban Arterial Corridor Scenario

This folder provides the input files and Python wrapper used to run the urban arterial corridor case in the XLO framework. The case is constructed from UAV-based field observations collected at three adjacent signalized intersections in Dongguan, China, and uses SUMO as the microscopic traffic simulator.

## Scenario description

UAV footage collected on 5 June 2025 at Dongguan, China, covers three adjacent signalized intersections. Vehicle trajectories were extracted, stitched and map-matched to an OpenStreetMap-derived network corrected against video. A 15-minute evening peak fragment (17:30–19:30) contains about 1,900 vehicles, including 60 trucks. Detector intervals were aligned with the 160-second signal cycle. The four metrics and WMAPE formulation remain unchanged, but the parameter space expands to 60 dimensions covering both passenger cars and trucks (the previous 29 plus vehicle length).

## Running SUMO manually

Before running the Python wrapper, first check whether the SUMO scenario can run directly.

If `new.rou.xml` has not been generated, run:

```bash
duarouter -c duarouter.conf
```

Then run the SUMO configuration file:

```bash
sumo -c test.sumocfg
```

If the command succeeds, SUMO will generate or overwrite the `output.___.xml` files.

## Running the Python workflow

After confirming that SUMO works, run:

```bash
python Running.py
```

The script will:

1. Load the observed metrics from `true_flow.csv`, `true_speed.csv`, `true_time.csv`, and `true_queue.csv`.
2. Update the calibrated passenger-car and truck parameters in the SUMO input files.
3. Start SUMO through TraCI using `test.sumocfg`.
4. Read the SUMO output XML files.
5. Compute four WMAPE-based calibration objectives.
6. Save the sampled inputs, objective values, and runtime records.

## Input and output files

### `Running.py`

Main Python entry point. It performs four tasks:

- edits the SUMO parameter settings;
- launches SUMO and controls the simulation through TraCI;
- parses the SUMO output files;
- computes the four WMAPE losses.

### `test.sumocfg`

Main SUMO configuration file. It specifies the network, route files, additional detector files, and simulation outputs used by this scenario.

### `taizhong_6.25.net.xml`

SUMO road network file for the urban arterial corridor. It defines the lanes, edges, connections, junction geometry, and traffic-light structure.

### `basic.vType.xml`

Vehicle type file. It stores passenger-car and truck behavioral parameters, including vehicle length, speed distribution, IDM car-following parameters, LC2013 lane-changing parameters, and junction-model parameters.

### `sumo_trips.xml`

Trip file generated from the processed UAV vehicle trajectories. It contains vehicle departure information and origin-destination information before route assignment.

### `new.rou.xml`

Route file used by SUMO. It contains the complete vehicle routes generated from `sumo_trips.xml` by the `duarouter` tool.

### `new.rou.alt.xml`

Alternative route file generated during route assignment. It is retained with the scenario resources for reproducibility.

### `duarouter.conf` and `duarouter_run.bat`

Configuration and Windows batch file for running `duarouter`, which converts trips into complete vehicle routes.

### `e1.add.xml`

E1 induction-loop detector file. E1 detectors are used to collect section-level traffic flow and speed.

### `e2.add.xml`

E2 lane-area detector file. E2 detectors are used to collect lane-area traffic states, especially queue-related indicators.

### `true_flow.csv`, `true_speed.csv`, `true_time.csv`, `true_queue.csv`

Ground-truth metrics extracted from the processed UAV trajectories. These files provide the observed traffic flow, speed, travel time, and queue length used for WMAPE calculation.

### `output.e1_out.xml`

SUMO E1 detector output. It records detector-level traffic counts and speed-related statistics and is used to evaluate simulated flow and speed.

### `output.e2_out.xml`

SUMO E2 detector output. It records lane-area measurements such as occupancy, speed, halting vehicles, and queue-related states.

### `output.queue.xml`

SUMO queue output. It records queue-related statistics during the simulation.

### `output.summary.xml`

SUMO summary output. It records network-level simulation statistics over time, such as loaded, running, waiting, arrived, and teleported vehicles.

### `output.tripinfo.xml`

SUMO trip information output. It records vehicle-level trip statistics, including departure, arrival, route length, duration, waiting time, and time loss.

## Objective function

The urban arterial corridor calibration is formulated as a four-objective minimization problem. The returned objectives are:

```python
[WMAPE_flow, WMAPE_speed, WMAPE_time, WMAPE_queue]
```

All objectives are WMAPE losses, so lower values are better.

The four simulated metrics are compared against the corresponding UAV-derived observations:

```text
traffic flow   -> true_flow.csv
speed          -> true_speed.csv
travel time    -> true_time.csv
queue length   -> true_queue.csv
```
