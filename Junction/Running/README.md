
# Isolated Urban Junction Scenario  
  
This folder provides the input files and Python wrapper used to run the isolated urban junction case in the XLO framework. The case is constructed from an open CitySim signalized-intersection scenario and uses SUMO as the microscopic traffic simulator.  
  
## Scenario description  
  
The CitySim dataset provided high-resolution vehicle trajectory data and a road network base map compatible with the microscopic traffic simulator SUMO. The 900-second simulation spans 17:40–17:55 on an evening peak hour and comprises about 1,300 vehicles. Vehicle trajectories were mapped to SUMO’s network using the sumolib library, assigning edges and positions based on geospatial coordinates. From these matches, we generated a trips file containing route information, departure and arrival times, then computed shortest paths with the duarouter tool. Lane-area and induction-loop detectors were deployed, with a 185-second aggregation interval matching the traffic signal cycle. Ground-truth metrics (flow, speed, travel time, and queue length) were compared to simulation outputs using WMAPE, yielding a 4-dimensional objective space. Five parameter families were considered—vehicle characteristics, speed distribution, car-following, lane-changing and junction model. Owing to the negligible share of trucks, we retain only the last four for passenger cars, resulting in 29 decision variables. The Intelligent Driver Model and the default LC2013 model in SUMO were adopted for car-following and lane-changing, respectively.

## Running SUMO manually  
  
Before running the Python wrapper, first check whether the SUMO scenario can run directly.  
  
If `new.rou.xml` has not been generated, run:  
  
```bash  
duarouter -c duarouter.conf```  
  
Then run the SUMO configuration file:  
  
```bash  
sumo -c test.sumocfg```  
  
If the command succeeds, SUMO will generate or overwrite the `output.___.xml` files.  
  
## Running the Python workflow  
  
After confirming that SUMO works, run:  
  
```bash  
python Running.py```  
  
The script will:  
  
1. Load the observed metrics from `true_flow.csv`, `true_speed.csv`, `true_time.csv`, and `true_queue.csv`.  
2. Update the calibrated passenger-car parameters in the SUMO input files.  
3. Start SUMO through TraCI using `test.sumocfg`.  
4. Control the signal phases and timings according to `IntersectionA-Signal.csv`.  
5. Read the SUMO output XML files.  
6. Compute four WMAPE-based calibration objectives.  
7. Save the sampled inputs, objective values, and runtime records.  
  
## Input and output files  
  
### `Running.py`  
  
Main Python entry point. It performs four tasks:  
  
- edits the SUMO parameter settings;  
- launches SUMO and controls the simulation through TraCI;  
- parses the SUMO output files;  
- computes the four WMAPE losses.  
  
### `test.sumocfg`  
  
Main SUMO configuration file. It specifies the network, route files, detector files, and simulation outputs used by this scenario.  
  
### `IntersectionA.net.xml`  
  
SUMO road network file for the isolated signalized junction. It defines the lanes, edges, connections, junction geometry, and traffic-light structure.  
  
### `IntersectionA-Signal.csv`  
  
Signal timing file. During simulation, `Running.py` uses TraCI to reproduce the real signal phase sequence and timing.  
  
### `basic.vType.xml`  
  
Vehicle type file. It stores passenger-car behavioral parameters, including speed distribution, IDM car-following parameters, LC2013 lane-changing parameters, and junction-model parameters.  
  
### `sumo_trips.xml`  
  
Trip file generated from CitySim vehicle trajectories. It contains vehicle departure information and origin-destination information before route assignment.  
  
### `sumo_flow.xml`  
  
Flow input file used for defining vehicle demand in SUMO.  
  
### `new.rou.xml`  
  
Route file used by SUMO. It contains the complete vehicle routes generated from `sumo_trips.xml` by the `duarouter` tool.  
  
### `new.rou.alt.xml`  
  
Alternative route file generated during route assignment. It is retained with the scenario resources for reproducibility.  
  
### `duarouter.conf` and `duarouter_run.bat`  
  
Configuration and Windows batch file for running `duarouter`, which converts trips into complete vehicle routes.  
  
### `loop.xml`  
  
E1 induction-loop detector file. E1 detectors are used to collect section-level traffic flow and speed.  
  
### `e2_det.xml`  
  
E2 lane-area detector file. E2 detectors are used to collect lane-area traffic states, especially queue-related indicators.  
  
### `e3_det.xml`  
  
E3 entry-exit detector file. E3 detectors are used to collect travel-time-related measurements between detector entry and exit points.  
  
### `true_flow.csv`, `true_speed.csv`, `true_time.csv`, `true_queue.csv`  
  
Ground-truth metrics extracted from the CitySim trajectory data. These files provide the observed traffic flow, speed, travel time, and queue length used for WMAPE calculation.  
  
### `true_timeLoss.csv`  
  
Observed time-loss reference retained with the scenario resources.  
  
### `output.e1_detector.xml`  
  
SUMO E1 detector output. It records detector-level traffic counts and speed-related statistics and is used to evaluate simulated flow and speed.  
  
### `output.e2_detector.xml`  
  
SUMO E2 detector output. It records lane-area measurements such as occupancy, speed, halting vehicles, and queue-related states.  
  
### `output.e3_detector.xml`  
  
SUMO E3 detector output. It records entry-exit detector measurements and supports travel-time evaluation.  
  
### `output.queue.xml`  
  
SUMO queue output. It records queue-related statistics during the simulation.  
  
### `output.summary.xml`  
  
SUMO summary output. It records network-level simulation statistics over time, such as loaded, running, waiting, arrived, and teleported vehicles.  
  
### `output.tripinfo.xml`  
  
SUMO trip information output. It records vehicle-level trip statistics, including departure, arrival, route length, duration, waiting time, and time loss.  
  
## Objective convention  
  
The isolated junction calibration is formulated as a four-objective minimization problem. The returned objectives are:  
  
```python  
[WMAPE_flow, WMAPE_speed, WMAPE_time, WMAPE_queue]  
```  
  
All objectives are WMAPE losses, so lower values are better.  
  
The four simulated metrics are compared against the corresponding CitySim-derived observations:  
  
```text  
traffic flow   -> true_flow.csv  
speed          -> true_speed.csv  
travel time    -> true_time.csv  
queue length   -> true_queue.csv  
```
